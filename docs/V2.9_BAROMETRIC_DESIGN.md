## 1. Objective

Add a barometric-pressure correction to the predicted **height of tide**, and
therefore to the computed access windows. The correction models the inverse
barometer effect: low pressure raises observed sea level above the
average-pressure prediction; high pressure depresses it.

UKHO predictions (and the harmonic model, which is calibrated to UKHO) assume
**average** barometric pressure. The correction adjusts for deviation from
that average using forecast pressure at each tide event's time.

This is distinct in purpose from the existing wind offset. The wind offset
adjusts the **effective ground/keel line** under a mooring (a threshold), not
the water height. The barometric correction adjusts the **water height**. The
two are orthogonal and compose.

---

## 2. Physics and coefficient

```
correction_m = clamp( (P_ref - P_event) * k * scale , ±max_correction_m )
```

| Quantity | Default | Source / rationale |
|---|---|---|
| `P_ref` reference pressure | 1013.25 hPa | Standard atmosphere; a documented placeholder. A location-specific average from Admiralty Sailing Directions could replace it, but see §6 — the regression that sets `k` absorbs any reference error into its intercept, so this value is not critical. |
| `k` coefficient | **0.0100 m/hPa** (adopted — see §6.5) | Empirically fitted from a 12-month Portsmouth gauge regression (wind-filtered ≈ 0.0100), coinciding with the textbook inverse-barometer 1 cm/hPa. The original UKHO-rounded prior was 0.00882 (34 hPa ≈ 0.30 m); the empirical value supersedes it. |
| `scale` | 1.0 | Manual trim for any local amplification. No empirical tuning source exists (see §6); stays 1.0. |
| `max_correction_m` clamp | 0.30 m | UKHO: pressure-driven change seldom exceeds 0.3 m. With the adopted `k=0.0100` the clamp trips at ±30 hPa from reference (±34 hPa under the original 0.00882 prior). Exceeding it implies bad data or a storm surge, which this correction does not model. |

**Sign convention (bidirectional):**

- P below reference (low barometer) → positive correction → **more** water than predicted.
- P above reference (high barometer) → negative correction → **less** water than predicted.

**Worked example (sets the deadband, §4.3):** at 1032 hPa, correction =
`(1013.25 − 1032) × 0.0100 = −0.19 m`. For a boat needing 3.3 m of a 4.2 m HW
(medium tide, range ~3 m), that shifts the window start ~13 min later and
the end ~18 min earlier — a window ~31 min shorter, in the conservative
direction. The shift grows for thresholds nearer HW (a deep-draught boat at
~3.9 m sees ~20/28 min) and can collapse marginal windows entirely.

## 3. Decision 1 — Where the correction enters

### 3.1 Events-list transform, mirroring the secondary-port offset

A new pure function `apply_barometric_correction(events, pressure_provider) ->
events` returns a **fresh copied list** of events with adjusted `height_m`,
applied **before** `compute_access_windows`. This mirrors
`secondary_port.apply_offset` ([app/secondary_port.py:66](../app/secondary_port.py)),
which is the existing precedent: a height-adjusting events transform applied
pre-window-computation, returning a copied list, resolving its constants via
cached config accessors.

Rationale:

- Keeps `compute_access_windows` ([app/access_calc.py:201](../app/access_calc.py))
  pure — the same separation the wind offset deliberately preserved by living
  in `compute_next_window_with_wind` rather than the core.
- Returning a copied list means the correction **never** touches stored rows
  (`tide_data`, `harmonic_predictions`), which keeps the harmonic-residual
  monitor and calibration scripts pressure-blind and uncontaminated.
- Single, testable unit. Uniform across UKHO and harmonic source paths.
- Composes with the wind offset on orthogonal axes: barometric shifts event
  **heights**; wind shifts the **threshold** (`safety_margin_m + wind_offset_m`).
  Both apply without interaction.

Apply order: **after** `secondary_port.apply_offset` (sample pressure at the
Langstone-corrected event time).

### 3.2 Per-mooring opt-in toggle + system master

Two gates; effective correction requires **all** of:

1. **System master** — `barometric.enabled` in `model_config.json` (default
   `false`). Rollout/kill switch.
2. **Per-mooring opt-in** — `barometric_enabled INTEGER DEFAULT 0` on
   `moorings`, mirroring `wind_offset_enabled`.
3. **Forecast available and fresh** for the event time (§4.4).

The correction's **value is system-level** (one pressure series, one `k`, one
reference — every opted-in mooring gets the identical shift at a given tide);
only the **opt-in decision is per-mooring**. This is a deliberate refinement of
the plan's "system-level, not per-mooring" line: the regional-physics rationale
holds for the data, and we add per-owner choice on top. It is a simpler gate
than wind (which also needs `shallow_direction` and `shallow_extra_depth_m > 0`,
because those are genuine per-mooring physics).

Consequences:

- The anonymous `/calculate` path has no mooring (`mooring is None`,
  [app/main.py:1311](../app/main.py)), so it always shows the **pure** forecast —
  backward-compat falls out for free.
- Default `0` gives backward compatibility and the plan's safe-rollout posture.
- It rides the existing `save_mooring` path, so it is PIN-gated as a
  state-changing op with no new gating logic; `_public_mooring` may expose it.
- Two skippers on the same tide can see different window times (opted-in vs
  not). Intended; document in README (the same situation wind-adjusted windows
  already create).

### 3.3 Grounding enumeration stays uncorrected (v2.9)

`ensure_wind_jobs_scheduled` ([app/scheduler.py:409](../app/scheduler.py)) keeps
using **uncorrected** heights to enumerate worst-case groundings. Correcting it
would make scheduled job times pressure-dependent (re-scheduling churn in a
second subsystem), and uncorrected = lower water = earlier worst-case grounding
= the conservative direction anyway. Only the windows that are *shown and fed*
are corrected.

## 4. Decision 2 — Forecast source, write cadence, churn control

### 4.1 Forecast source

The deployed OWM integration uses only the **current-weather** endpoint
(`/data/2.5/weather`, [app/wind.py:105](../app/wind.py)); `pressure_history` is
current-only and pruned to 24 h. There is **no forecast capability today** —
this is net-new work.

Use OWM's free **5-day / 3-hour forecast** (`/data/2.5/forecast`), which carries
`main.pressure` (prefer `main.sea_level` when present) per 3-hourly step (~40
steps). A `pressure_provider(target_time)` interpolates linearly between the two
bracketing forecast steps and reports the forecast's age for the staleness gate.

> **Verified 15 June 2026** against the live `tidal-access` container (probe
> using the deployment's own key): `/data/2.5/forecast` returns HTTP 200 on the
> existing free-tier key — 40 entries at 3-hour steps, ~117 h horizon (≈4.9
> days). `main.pressure` and `main.sea_level` are both populated on all 40
> entries and are equal (1014/1014) at the Langstone lat/lon, confirming OWM
> reports sea-level pressure here — the station-vs-sea-level concern is moot.
> Pressure is integer hPa (~0.009 m resolution, well below display rounding).

### 4.2 Revision cadence: piggyback the daily job

Fetch the forecast and regenerate feeds once daily in the existing 02:00 job
([daily_ukho_fetch](../app/scheduler.py)), where UKHO and harmonic already
refresh. Do **not** drive feed writes from the 15-minute conditions loop. This
respects the "feed mutated in exactly three places" invariant: the correction
rides the daily job, the manual "Update Feed", and (for one window) the wind
grounding job — it is not a new write path. The read-path `/calculate` reads the
stored daily forecast (a future tide's forecast is the same regardless of which
hour of the day you read it).

### 4.3 Churn control — one mechanism per feed type

**Standalone corrected tide feed — rounding is the implicit deadband.** A
barometric correction is a near-uniform vertical shift, so on a tide-event feed
it changes HW/LW **heights but not their times**. Heights render at 0.1 m
([app/ical_manager.py:614](../app/ical_manager.py)) under stable cycle-based
UIDs. Regenerate wholesale daily; the `.ics` is byte-identical unless a
correction crosses a 0.1 m boundary, in which case only the height text changes
on a same-time, same-UID event. No extra logic.

**Per-mooring access-window feeds — explicit minutes deadband.** A height
correction moves the threshold-crossing **times**. Store windows full-precision;
when regenerating, only rewrite an event if a window edge moved **≥ 5 min**
versus the currently-stored raw edge (`barometric.window_deadband_minutes`).
Rationale: the model's own intrinsic timing uncertainty is ~15 min HW / ~19 min
LW ([model_config.json:33](../app/model_config.json)), so a 5-min deadband sits
below the noise floor. The deadband is compared on the **raw** edge (not the
rounded display value) so an edge parked near a rounding boundary cannot flicker
between adjacent 5-min slots on sub-minute forecast jitter.

Result: churn tracks materiality. Small/ordinary forecast revisions (≤ ~5–6 hPa
day-over-day) never rewrite; a genuine synoptic change (≥ ~8–10 hPa) propagates
and converges as the event approaches. **No "freeze near the event"** — that was
rejected on safety grounds (a late fast-moving low is exactly when the latest
correction matters most); the deadband plus forecast convergence stabilise the
near-term answer naturally.

### 4.4 Forecast horizon and staleness

- **Horizon is a data boundary, not a tuning knob.** Correct each event with the
  best-available forecast; beyond the forecast's reach (~5 days) apply **no**
  correction (never extrapolate stale pressure). The 7-day feeds are therefore
  corrected for ~days 0–5 and identical to baseline for ~days 5–7; a
  harmonic-sourced per-mooring feed is corrected only in its first ~5 days.
- **Staleness guard is per-event.** Disable correction for an event only when no
  usable forecast covers its time, and revert that event to baseline (logged) —
  never a wholesale revert that churns the entire feed on one failed fetch.
- **Staleness threshold: `barometric.forecast_staleness_hours`, default 36 h.**
  *Revised from the 6 h discussed earlier.* 6 h suited a current-pressure model;
  this correction uses a **daily-refreshed forecast** (up to ~24 h old by the
  next run), so 36 h tolerates one missed daily fetch before the forecast is
  treated as stale.

### 4.5 Read path stays always-live

`/calculate` applies the latest correction with **no deadband** — it is a
one-shot, in-the-moment computation and should show the freshest estimate. The
intended consequence is that the calculate view can differ slightly from the
subscribed feed (which lags behind the deadband). Document as designed.

### 4.6 Conservative 5-minute rounding of all displayed windows

Independent of barometric: **all** access-window edges shown to users (vessel
and tender; corrected for wind, pressure, or neither) are rounded to the nearest
5 minutes to avoid false precision, since the model's own timing uncertainty is
~15–20 min.

- **Mode: conservative inward.** Start → round **up** (later); end → round
  **down** (earlier). Both bounds move toward HW; the window always shrinks and
  never overstates access. (`window_display.rounding_mode = "conservative_inward"`,
  `window_display.rounding_minutes = 5`.)
- **Applied at render only** — the API response serialization and iCal
  rendering, via one shared helper so they cannot drift. Computation, storage,
  the deadband comparison, and the calibration scripts all keep full precision.
- **Displayed duration recomputed** from the rounded edges.
- **Degenerate collapse:** when conservative rounding yields `start ≥ end` (a
  sub-grid window on a marginal tide), collapse to a "negligible — no usable
  access" state; never emit an inverted or zero-length `.ics` event. This is the
  conservative outcome — a <5 min window is not actionable.
- **Scope:** bounded windows only. The special states are untouched —
  `below_threshold` (no times), `always_accessible` (an LW-to-LW span shown as a
  state), `wind_no_access` (an at-HW marker), `incomplete_data` — as is the HW
  event time/height (a tide prediction, not a window edge).

With deadband-on-raw + round-on-render, a non-rewrite never changes the
displayed slot and any rewrite (≥5 min raw move) always shifts it by at least
one visible slot. Clean: no flicker, no invisible rewrites.

## 6. Decision 3 — Validation and tuning

### 6.1 The harmonic-residual monitor cannot validate this

`compute_harmonic_residuals` ([app/database.py:1451](../app/database.py))
compares **harmonic predictions to UKHO events** — both average-pressure
predictions. Neither carries a barometric signal, so stratifying their
difference by pressure reveals model error, not correction quality. The plan's
"validation #1" (stratify the existing monitor by pressure) is misaimed and is
**not** used. A barometric signal appears only when a prediction is compared to
a **real measured water level**.

### 6.2 BODC Portsmouth is the sole and sufficient validator

The only measured-water-level source available is the UK National Tide Gauge
Network at Portsmouth.

- **Programmatic access confirmed — no manual BODC download needed.** Portsmouth
  is station `E71839` (mAOD) on the Defra/EA flood-monitoring service. The live
  **Tide Gauge API** serves a rolling ~4 weeks (clean REST/JSON, no key), and the
  flood-monitoring **archive** (`…/archive/readings-YYYY-MM-DD.csv`, daily
  all-station dumps back to ~2016) was verified to include the Portsmouth
  15-minute tidal rows — so `scripts/validate_barometric_k.py --source ea-archive`
  pulls multi-year history directly, filtering ~96 Portsmouth rows per daily
  dump. The **BODC** quality-controlled archive (15-min from 1993, manual
  download) remains an optional gold-standard alternative via `--source csv`.
- **Sufficient for tuning `k`, and better than in-situ data:** the archive
  already spans every pressure regime, so `k` can be fit **now**, before
  enabling the feature.
- **Confounds fall into the regression intercept, not the slope.** Fit
  `(measured − astronomical_prediction) = intercept + slope · (P_ref − P)`. The
  **slope is the empirical `k`**. Datum offset (Portsmouth CD ↔ OD Newlyn) and
  instrument bias are constants → intercept. So `k` is identifiable without
  precise datum alignment.
- **Two caveats:** (1) storm surge co-varies with low pressure (deep lows bring
  wind setup), which can bias `k` high on the low tail — mitigate by the
  regression and optionally excluding high-wind days; (2) it is Portsmouth, not
  Langstone — see §6.3.

### 6.3 No Langstone-specific tuning; `scale_factor` stays 1.0

Per project knowledge, mooring observations will **not** realistically
accumulate (one user, too infrequent), so they are **not** a tuning source. This
does not matter, because the inverse-barometer effect is **regional**: pressure
systems are ~1000 km across and UKHO notes water level "responds to average
pressure over a considerable area", so the coefficient is essentially uniform
over the ~6 km between Portsmouth and Langstone. This is unlike the **tidal**
secondary-port offset, which captures genuine local astronomical-tide
propagation. Therefore:

- A Portsmouth-derived `k` is the correct `k` for Langstone, to better than our
  precision.
- `barometric.scale_factor` stays **1.0**, kept only as a documented manual
  override with no empirical tuning source.
- **No observation-pipeline or schema changes in v2.9** (the transition-timestamp
  and per-observation pressure-snapshot additions the plan recommended are
  dropped — they were justified only as a tuning source that will not deliver).

### 6.4 `k` is a stable physical constant

Unlike harmonic constituents (which can drift and need periodic recalibration),
the inverse-barometer coefficient is physics and does not drift. So:

- Validation is a **one-off offline fit** against the BODC archive (a `scripts/`
  tool), done before enabling.
- No ongoing automated validation loop, no `sea_level_observations` table, no
  daily gauge fetch — unless live monitoring is later wanted.
- Keep only **lightweight operational logging** of the applied correction
  (pressure used, raw/clamped value, source, guard/clamp fires) for transparency
  and debugging — explicitly **not** a tuning dataset.

### 6.5 Empirical result and coefficient decision (15 June 2026)

A 12-month fit (`scripts/validate_barometric_k.py --source ea-archive`,
2025-06-01..2026-05-31; Portsmouth gauge E71839 mAOD vs harmonic, ERA5
pressure) over 32,938 matched readings spanning 969.7–1034.2 hPa gave:

- All points: **k = +0.01047 m/hPa** (SE 0.00014), R² 0.14, intercept −2.59 m.
- Wind ≤ 8 m/s: **k = +0.00999 m/hPa** (SE 0.00015) — the cleaner estimate
  (excluding surge-contaminated high-wind periods lowers k, as expected).

The standard error is tiny and the pressure range wide, so this is a robust
empirical estimate. The wind-filtered **k ≈ 0.0100** lands on the textbook
inverse-barometer 1 cm/hPa and is ~13% above the UKHO-rounded prior 0.00882.

**Decision — CLOSED 15 June 2026: ship k = 0.0100.** A second year of EA data
was judged not worth it: the fit is already **systematic-limited** (the
all-points-vs-wind-filtered spread ~0.0005 exceeds the SE 0.00014 several-fold,
so more data shrinks the wrong term); **k is a stable physical constant** not
expected to vary year to year; the **pressure range is already near-maximal**
(64.5 hPa, nearly the full ±34 hPa clamp); and the 13% gap from the old prior is
**below the 5-min display rounding** while the deployed error budget is dominated
by **forecast-pressure error** (a few hPa → few cm) and the **harmonic/curve
model** (~0.2 m RMS — R² was only 0.14). The chosen follow-up is the production
**self-consistency check** (does the corrected residual still slope with pressure
once live?), not more offline fitting. Caveats retained: single year; EA
telemetry is less quality-controlled than BODC; the instantaneous wind filter
leaves some surge, so the true value could sit marginally below 0.0100. A larger
k slightly increases the non-conservative (window-widening) low-pressure
correction, but the 0.30 m clamp bounds the worst case.

## 8. Backward compatibility, failure modes, what it does NOT do

- **Flag off / mooring not opted in / no fresh forecast → exact pre-v2.9
  behaviour.** Lenient fallback: a missing/malformed `barometric` section means
  the feature is off.
- **Clamp** at `max_correction_m`; log a warning when it fires (anomalous data
  or surge).
- **It does not** model storm surge (wind-driven, can exceed the static range),
  the lag between a pressure change and the water response, and **does not**
  replace the wind offset (distinct mechanism; both may apply, sanity-check
  combined magnitude).

## 10. Build plan — session-sized chunks

Each chunk is scoped to a single bespoke session: self-contained, independently
verifiable, and leaving the branch in a working, rebuildable state. The feature
stays dark (`enabled=false`) until Session I. There is no test suite —
verification is syntax check, the calibration/probe scripts, and manual UI
exercise at `http://localhost:8866` (rebuild for `app/` changes:
`docker compose up -d --build`).

Dependencies and parallelism are noted; **B, C, and H have no dependency on each
other** and can be slotted in any order.

### Session A — Config scaffolding + correction core *(no deps)*
- Add the `barometric` and `window_display` sections to `model_config.json`
  (master `false`).
- Add cached accessors in `app/config.py` for each key (lenient pattern).
- New `app/barometric.py`: `apply_barometric_correction(events, pressure_provider)`
  — pure, mirrors `secondary_port.apply_offset`; per-event clamp + sign;
  uncorrected pass-through for events with no/stale pressure.
- **Done when:** syntax check passes; a `python -c` / tiny probe exercises the
  function with a fake provider (high/low/missing pressure) and returns the
  expected shifts. Called nowhere yet → zero behaviour change.

### Session B — Pressure forecast fetch + storage + provider *(deps: A)*
- `fetch_pressure_forecast()` in `app/wind.py` (OWM `/data/2.5/forecast`; prefer
  `main.sea_level`).
- `pressure_forecast` table in `SCHEMA` + store/get/cleanup in `app/database.py`
  (latest-fetch-wins per target time; prune old).
- `pressure_provider(target_time)` → interpolated pressure + age, or None.
- Add the daily forecast fetch to `daily_ukho_fetch` (store only; apply nowhere).
- **Done when:** `docker exec` probe fetches and stores forecast rows; provider
  returns sensible values and degrades to None past the horizon / on failure.
- **Verified 15 June 2026** (see §4.1): the endpoint is reachable on the
  existing key, 3-hourly / ~117 h horizon, pressure fully populated as SLP.

### Session C — Universal conservative 5-min rounding *(no deps; touches all users)*
- Shared `round_window_conservative` helper + `window_display` config.
- Apply at the `/calculate` response and iCal rendering; recompute duration;
  degenerate-collapse to no-access; leave special states and HW times alone.
- **Done when:** UI and a downloaded `.ics` show edges on the 5-min grid, windows
  never overstate, short windows collapse cleanly, durations match the rounded
  edges. (Independent of barometric — verifiable with the feature off.)

### Session D — Read-path wiring + per-mooring toggle + UI *(deps: A, B)*
- `barometric_enabled INTEGER DEFAULT 0` on `moorings` (startup `ALTER TABLE`);
  thread through `save_mooring` and `_public_mooring`.
- In `/calculate`: between events build and `compute_access_windows`, gate on
  (master ∧ `mooring.barometric_enabled` ∧ fresh provider) → correct events;
  tender pass inherits via the shared list.
- Toggle in the mooring config panel in `app/static/index.html` (beside wind).
- **Done when:** with master on, a toggled-on mooring's calculate shows shifted
  windows vs off; anonymous calculate stays pure; PIN gating unaffected. (Rebuild
  for static + Python.)

### Session E — Feed write path + deadband + wind-job composition *(deps: A, B, D)*
- In `daily_ukho_fetch`'s per-mooring loop and the manual feed-update path:
  correct events for opted-in moorings before `compute_access_windows`.
- Deadband: only rewrite a `calendar_events` row if a raw edge moved
  ≥ `window_deadband_minutes` vs the stored raw edge.
- `wind_observation_job`: feed corrected events for opted-in moorings (one
  window).
- Per-event staleness handling + baseline revert (logged).
- **Done when:** a probed daily run with a stored forecast writes corrected
  windows for opted-in moorings only; re-running with a sub-5-min forecast nudge
  produces no rewrite; a ≥5-min change does.

### Session F — Standalone pressure-corrected tide feed *(deps: A, B)*
- `generate_langstone_ukho_7d_pressure_corrected_feed()` (heights corrected,

- **Done when:** the new feed validates as iCal, heights differ from the

### Session G — Operational logging + conditions surfacing *(deps: D, E)*

- **Done when:** the activity log shows per-run barometric entries; clamp/staleness
  events are visible.

### Session H — BODC offline k-validation script — **DONE 15 June 2026**
- `scripts/validate_barometric_k.py` regresses measured sea level − harmonic
  prediction on (P_ref − P); slope = empirical k. Two input modes: `--source ea`
  (Portsmouth gauge E71839 mAOD via the Defra/EA flood-monitoring API, rolling
  ~4 weeks — pipeline check only) and `--source csv` (the definitive multi-year
  BODC archive). Historical pressure + wind come from the Open-Meteo ERA5
  archive (free, no key), interpolated to each reading. Pure-Python OLS (no
  numpy). BODC CSV format documented in
